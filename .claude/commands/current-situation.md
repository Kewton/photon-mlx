# 不具合事象の整理とIssue登録

## 概要
ユーザーとの会話で明らかになった不具合事象を正確に整理し、再現可能な形でGitHub Issueに登録します。会話履歴から事象・再現手順・影響範囲を抽出し、構造化されたバグレポートを作成します。

## 使用方法
```
/current-situation [対象の概要説明]
```

例：
```
/current-situation file.editが大きなファイルで失敗する件について
/current-situation ANVIL_FINALが早期発火する問題
```

## 前提条件
- ユーザーとの会話で不具合の内容がある程度明確になっていること
- GitHubリポジトリ（https://github.com/Kewton/photon-mlx）にアクセス可能
- `gh` CLIが認証済み

## 実行内容

あなたは品質保証エンジニアとして、不具合事象を正確に整理しIssueに登録します。

### Step 1: 会話履歴からの事象抽出

現在の会話履歴を分析し、以下の情報を抽出・整理します：

#### 抽出項目

| 項目 | 説明 |
|------|------|
| **事象の要約** | 何が起きているか（1-2文） |
| **期待される動作** | 本来どう動くべきか |
| **実際の動作** | 実際にどう動いたか |
| **再現手順** | 事象を再現するための具体的な手順 |
| **再現頻度** | 必ず発生 / 高頻度 / 時々 / 不明 |
| **影響範囲** | どの機能・ユーザーに影響するか |
| **環境情報** | OS, モデル, PHOTON-RepoRAGバージョン, 設定等 |

### Step 2: 関連ログの収集

会話中に言及された実行ログ、エラーメッセージ、stderr出力を収集します。

```bash
# 直近のPHOTON-RepoRAG実行ログがあれば取得
ls -la .photon-mlx/logs/ 2>/dev/null | tail -5

# セッション情報
ls -la .photon-mlx/sessions/ 2>/dev/null | tail -5
```

また、関連するソースコードの箇所を特定します：

```bash
# 事象に関連するソースファイルを検索
# （事象のキーワードに基づいてgrepする）
```

### Step 3: 既存Issueとの重複チェック

```bash
# 類似のオープンIssueがないか確認
gh issue list --repo Kewton/PHOTON-RepoRAG --state open --json number,title --limit 30
```

類似Issueが存在する場合はユーザーに報告し、新規作成か既存への追記かを確認します。

### Step 4: Issue本文の作成

以下のテンプレートで整理します：

```markdown
## 事象
${事象の要約}

## 再現手順
1. ${具体的な手順1}
2. ${具体的な手順2}
3. ...

## 期待される動作
${本来の動作}

## 実際の動作
${実際に起きたこと}

## 再現頻度
${必ず / 高頻度 / 時々 / 未確認}

## 環境情報
- **OS**: ${OS情報}
- **PHOTON-RepoRAGバージョン**: ${バージョン}
- **モデル**: ${使用モデル}
- **設定**: ${関連する設定値}

## ログ・エビデンス

<details>
<summary>実行ログ</summary>

${ログ内容}

</details>

## 影響範囲
${影響するモジュール・機能}

## 関連するソースコード
${事象に関連するファイル・関数の一覧}

## 備考
${会話中に判明したその他の情報}
```

### Step 5: ユーザー確認

作成したIssue本文をユーザーに提示し、確認を求めます：

```
以下の内容でバグIssueを作成します。修正・追加はありますか？

タイトル: [Bug] ${タイトル}

${Issue本文のプレビュー}

(y: 作成 / 修正内容があれば入力してください)
```

### Step 6: Issue登録

```bash
gh issue create --repo Kewton/PHOTON-RepoRAG \
  --title "[Bug] ${タイトル}" \
  --label "bug" \
  --body "${Issue本文}"
```

### Step 7: 結果報告

```
Issue #${NUMBER} を作成しました: ${タイトル}
URL: ${Issue URL}

次のアクション:
- /CauseAnalysis ${NUMBER} で根本原因分析を実行
- /bug-fix ${NUMBER} でバグ修正を開始
```

## 品質基準

作成するIssueは以下を満たすこと：

- [ ] 事象が第三者にも理解できる明確な記述
- [ ] 再現手順が具体的で実行可能
- [ ] 期待動作と実際動作の差分が明確
- [ ] 関連ログ・エビデンスが含まれている
- [ ] 影響範囲が特定されている
- [ ] `bug` ラベルが付与されている
- [ ] 既存Issueとの重複がないことを確認済み

## エラーハンドリング

| エラー | 対応 |
|--------|------|
| 会話履歴に不具合情報が不足 | ユーザーに追加情報を質問 |
| 再現手順が不明確 | ユーザーと対話して具体化 |
| 類似Issueが存在 | ユーザーに新規作成か既存追記かを確認 |
| gh CLI認証エラー | 認証手順を案内 |
