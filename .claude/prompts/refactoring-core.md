# リファクタリングコアプロンプト

## 原則

外部から見た動作を変えずに、内部構造を改善する。

**鉄則**: テストが壊れたら即座にロールバック。

## SOLID原則（Rust適用）

### S - 単一責任の原則
- 各モジュール・構造体は1つの責務のみを持つ
- ファイルが長すぎる場合はモジュール分割を検討

### O - 開放閉鎖の原則
- トレイトを使って拡張可能な設計にする
- 新機能追加時に既存コードの変更が最小限になるように

### L - リスコフの置換原則
- トレイト実装は契約を守る
- デフォルト実装は安全な動作を保証

### I - インターフェース分離の原則
- トレイトは小さく保つ
- 不要なメソッドを実装させない

### D - 依存性逆転の原則
- 具体型ではなくトレイトに依存する
- テスタビリティを重視

## Rustイディオム

### 所有権とライフタイム
- 不必要な `clone()` を排除
- 借用で十分な場合は所有権を取らない
- ライフタイムは明示的に（省略できる場合でも可読性重視）

### パターンマッチ
- `if let` / `while let` を活用
- `match` の網羅性を利用したコンパイル時チェック

### イテレータ
- `for` ループより `iter()` チェーンを優先
- `collect()` の型注釈を明確に

### エラーハンドリング
- `unwrap()` → `?` 演算子 or 適切なエラー処理
- カスタムエラー型の導入を検討
- `anyhow` / `thiserror` の活用

### 型システムの活用
- ニュータイプパターンで意味を付与
- `enum` で状態を明示
- `Option<T>` で null安全性を確保

## リファクタリング手順

### 1. 現状把握

```bash
ruff check . 2>&1
python -m pytest torch_ref/tests/ photon_mlx/tests/ -v
```

### 2. 改善点の特定

優先順位：
1. コンパイラ警告・ruff指摘
2. `unwrap()` の除去
3. 重複コードの統合（DRY）
4. 関数の分割（長すぎる関数）
5. モジュール構造の整理
6. 型安全性の向上

### 3. 小さな変更を繰り返す

1つの変更ごとに：

```bash
python -m pytest torch_ref/tests/ photon_mlx/tests/ -v
ruff check .
```

### 4. 最終検証

```bash
ruff format --check .
ruff check .
python -m pytest torch_ref/tests/ photon_mlx/tests/ -v
python -m pytest
```
